  • Analysis Description

      All aspects of the ChocAn Data Processing system will be computerized as well as each interface with the ChocAn system. The data will be stored on highly advanced data center servers located on the premises of the ChocAn Company data center. The providers, operators, managers, and other authorized ChocAn agents will use third-party provided hardware platforms as terminals on which to communicate with members, providers, and the ChocAn data center servers. The ChocAn Processing System will use client/server protocols to receive and transfer data from each of the terminal sources.

  • Assumptions

    • Prospective & current members only pay ChocAn by using a credit card
    • HCP-Terminals Determine how to Display data messages retrieved from the ChocAn Data Processing System
    • ChocAn Members only have access to ChocAn Benifits during normal business hours (9:00am-5:00pm)
    • HCPs only use the members' cards with the intent of providing services that are subsequently billed to ChocAn
    • HCPs only bills ChocAn for validated members of whom recieved Provider services
    • HCPs do not return the member card to its holder until after they have sucessfully billed ChocAn
    • HCPs know that the card must be swiped a second time to enter "Billing Mode"
    • All billing requests are made during normal hours of business operations on the same day as the service request
    • HCPs use their own Forms which is not part of the software system in order to keep track of their bookkeeping as it relates to ChocAn Billable Transactions
    • HCPs insert the MM-DD-YYYY format correctly
    • ChocAn System never has incomplete, pending transactions before losing power
    • Managed Provider Sessions are always closed properly and do not need to handle special exceptions

  • Actors and Use Cases

    Use Case: [ Manage Session ]

    Context: A Health Care Provider (HCP) uses a Special Terminal (HCP-T) which requires authentication for sessions.

    Actors: HCP Terminal (HCP-T)

    Main Success Scenario:

    • 1.0 The system receives notification that an HCP-T needs an authenticated session
    • 1.1 The system sends a message to the HCP-T requesting the provider number
    • 2.0 The system recieves a Provider Number and authenticates a Provider Session
    • 2.0 The system sends a message to the HCP-T confirming session authentication
    • 3.0 The system receives notification to close an HCP-T authenticated session
    • 3.1 The system closes authenticated session for a particular HCP-T

    Extensions:

    • 1.A.0 The system rejects the Provider Number
    • 1.A.2 The system continues at Use Case 1.1
    • 1.B.0 The system stops receiving notification that an HCP-T needs an authenticated session
    • 1.A.2 The Use Case Ends
    Use Case: [ Process Service Requests ]

    Context: A Health Care Provider (HCP) uses a Special Terminal (HCP-T) which validates a requesting customer's ChocAn membership status prior to an HCP's supplying a particular service.

    Actors: ChocAn Member, HCP Terminal (HCP-T)

    Main Success Scenario:

    • 1.0 The system receives a processing request (a call) from an HCP Terminal with encoded member information
    • 1.1 The system collects the Current Date and Time (MM-DD-YYYY HH:MM:SS)
    • 1.2 The system collects the Health Care Providers Number based on client HCP-T
    • 1.3 The system collects the Member's number
    • 2.0 The system authenticates the ChocAn member as valid
    • 2.1 The ChocAn system sends a "Valid Member" message to the HCP Terminal for display

    Extensions:

    • 2.A.0 The system rejects member authentication and finds the member is suspended.
    • 2.A.1 The ChocAn system sends a "Suspended member" message to the HCP Terminal for display
    • 2.B.0 The system rejects member authentication and finds the member number is invalid.
    • 2.B.1 The ChocAn system sends an "Invalid Member number" message to the HCP Terminal for display
    Use Case: [ Process Billing Requests ]

    Context: A Health Care Provider (HCP) uses a Special Terminal (HCP-T) which allows the provider to submit billing requests for services rendered.

    Actors: HCP Terminal (HCP-T)

    Main Success Scenario:

    • 1.0 The system receives and Validates a Billing request from an HCP-T
    • 1.1 The system sends the text "Validated" to the HCP-T for display
    • 1.2 The system prompts the HCP-T to retrieve the date in the following format: MM-DD-YYYY
    • 2.0 The system receives a date from an HCP-T in the format MM-DD-YYYY
    • 2.1 The system collects the Date the service was provided
    • 2.2 The system prompts the HCP-T to retrieve the six-digit code corresponding to a service
    • 3.0 The system receives a valid six-digit code corresponding to a service
    • 3.1 The system returns the name of the service to the HCP-T display
    • 3.2 The system Prompts the HCP-T to confirm the correct service was selected
    • 4.0 The system receives confirmation that the service name displayed is the service provided
    • 4.1 The system collects the Service Code
    • 4.2 The system prompts the HCP-T to retrieve Entered Comments or confirmation to Apply Billing Processing
    • 5.0 The system receives a request to enter additional comments
    • 5.1 The system prompts the HCP-T to insert Comments limited to 100 characters
    • 5.2 The system receives and collects the submitted comments from HCP-T
    • 6.0 The system takes the collected data and writes it to the "File of Services Provided" on the ChocAn Datacenter Disks
    • 6.1 The system sends a validation, "Billing Applied," message to the HCP-T for display.
    • 6.2 The system sends the Fee of the submitted services to the HCP-T for Display
    • 6.3 The system sends a "Transaction Completed" message to the HCP-T for Display
    • 7.0 End this Use Case

    Extensions:

    • 1.A.0 The system fails to validate billing request
    • 1.A.1 The ChocAn system sends an invalide message to the HCP Terminal for display
    • 2.A.0 The system does not recieve a valid date format
    • 2.A.1 The system continues at Use Case 2.0
    • 3.A.0 The system rejects the six-digit code recieved from the HCP-T
    • 3.A.1 The ChocAn system sends a error message to the HCP Terminal for display
    • 3.A.2 The system continues at Use Case 3.0
    • 3.B.0 The system recieves a message to cancel the billing process
    • 3.A.2 The system continues at Use Case 7.0
    • 4.A.0 The system receives denial that the service name displayed is the service provided
    • 4.A.1 The system continues at Use Case 3.0
    • 5.A.0 The system receives request to apply billing procedures
    • 5.A.1 The system continues at Use Case 6.0
    Use Case: [ Request Provider Directory ]

    Context: A Health Care Provider (HCP) uses a Special Terminal (HCP-T) to access the Provider Directory.

    Use Case: [ File of Services Provided ]

    Context: List of services performed each week is produced and included in Member Report and Provider Report.

    Actors: Member, Provider

    Main Success Scenario:

    • 1.0 After each service is performed, a record of it is taken by the provider.
    • 2.0 When member report is generated, this list is added to it.

    Use Case: [ Member Report ]

    Context: At the end of each work week, specifically midnight, the main procedure for accounting runs at ChocAn's Data Center. It stores and prints the week's services to numerous reports. If a member consults with one of ChocAn's providers, at any time of the week, he/she is e-mailed a list of services that are relative to him/her, sorted in a chronological order of service date. A specific amount of data is required in each e-mail given to the member.

    Actors: Manager, Member

    Main Success Scenario:

    • 1.0 On Friday at midnight, the accounting procedure is ran at the Data Center.
    • 2.0 The current week's file of services provided is read and prints various reports.
    • 2.1 The manager has the ability to run each report individually at any given time during the week, if not, proceed to 3.0.
    • 3.0 Every member that requested services that week from a provider is e-mailed a list of services provided, arranged by order of service's date. The e-mailed report consists of: Member's name (25 characters), Member's number (9 digits), Member's resident address (25 characters), Member's city (14 characters), Member's state (2 letters), and the Member's ZIP code (5 digits). For all of the services that the specific member requested, there must be a Date of Service (MM-DD-YYYY), Provider's name (25 characters), and the Service name (20 characters).

    Extensions:

    • 3.A.0 If the Data Center cannot find the correct e-mail of its member, then the manager will be notified and must request that the member tell his/her updated e-mail address.
    • 3.B.0 If any of the components in the specific member's e-mail is missing or input validity is wrong, the manager is informed and must request a new report be e-mailed after the changes are implemented.
    • 3.C.0 Once the member receives his/her e-mail and if any of the data is incorrect, then the member has the ability to send a response to ChocAn's manager to have the data resolved and resent to his/her e-mail.
    Use Case: [ Provider Report ]

    Context: For all the providers that billed ChocAn for the current week, there are sent a report via e-mail containing the services that he/she provided to members. At the end of the providers' reports, there is a summary of that contains the overall number of consultations that he/she preformed with members of ChocAn, the total fee for the current week, and other provider based information that is relative to each provider.

    Actors: Provider, Manager

    Main Success Scenario:

    • 1.0 Each of ChocAn's providers that preformed a service is e-mailed as a report as an attachment, which states what services he/she preformed that week; For simplicity, the report has the same information as the provider's form, in the same order that the computer received the data.
    • 2.0 For each of the services that the provider preformed, this data is stored for each action: Date of Service (MM-DD-YYYY), the date and time that the data was received by ChocAn's computer (MM-DD-YYYY, HH:MM:SS), the member's name that requested the service (25 character), the member's ID number (9 digits), the selected service code (6 digits), and the required fee based on the service provided (up to $999.99), which is e-mailed to the provider in 3.0.
    • 3.0 At the end of said report, there is a summary that consists of the number of consultations with ChocAn's members and the total fee owed for that week; The fields of the provider's report include: Provider's name (25 characters), Provider's ID number (9 digits), Provider's street address (25 characters), Provider's city (14 characters), Provider's state (2 letters), Provider's ZIP code (5 digits), cumulative number of consultations with members (3 digits), and the total fee for the currrent week, this data is stored in said e-mail attachment.

    Extensions:

    • 3.A.0 If the report is incorrect or missing any information about the provider or the member, then the manager is informed and the e-mail is not sent until the information error is resolved.
    • 3.B.0 Once the provider receives his/her e-mail and if any of the data is incorrect, then the provider has the ability to send a response to ChocAn's manager to have the data resolved and resent to his/her e-mail.
    Use Case: [ Summary Report ]

    Context: At the end of each week, a summary report is printed and sent to ChocAn's manager for payment purposes. The report lists numerous things about each provider at ChocAn such as: number of consultations and services preformed throughtout the given week, and also total fees for each individual provider. Likewise at the end of the report, it prints a cumulative representation of the week regarding all of ChocAn's providers and how they contributed each week.

    Actors: Manager

    Main Success Scenario:

    • 1.0 A report is given to ChocAn's manager to verify account payments for each week; the content of said report consists of: every provider that preformed a service and needs to the be paid, the total number of consultations that each provider preformed, each of the providers' weekly total fee, the total number of providers that preformed services that current week, the overall total of consulations, and the total amount of fees for the current week.

    Extensions:

    • 1.A.0 If the report does not contain the previous information, then the manager has authorization to demand the data be input correct and maybe manually and then reprinted with the correct data.
    • 1.B.0 If there is not provider that preformed a service to a member for a current week, then there must be a message to the manager stating that there was not any services exchanged.
    Use Case: [ Manage Members ]

    Context: During the day, the software at the ChocAn Data Center is run in interactive mode to allow operators to add, remove, and modify member records. If a new client wants to become a member, an operator will add them into the system.

    Actors: Operators, Client

    Main Success Scenario:

    • 1.0 During the day, an operator at the ChocAn Data Center can add members, remove members, and update members’ reports. The operator will select what to action to take: add, update, or removing a member. Depending on the selection made, the action will be executed. If a new client wants to become a member, an operator will add them into the system.

    Extensions:

    • 1.A.0 If the add member option is selected, the operator will add a new member into the ChocAn Data Center.
    • 1.B.0 If the remove member option is selected, the operator will remove an existing member from the ChocAn Data Center.
    • 1.C.0 If the update member records option is selected, the operator will modify an existing member’s records in the ChocAn Data Center.
